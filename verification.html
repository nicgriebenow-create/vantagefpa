Verification

Anyone can generate a deck now. Verification is the product.

The hard part is standing behind a number when the other side of the table pushes. This page describes how a Vantage figure gets produced, what checks it clears before anyone sees it, and how you check the work yourself without taking a word of this on trust.

The Architecture

Three things hold the output up

Most finance work is verified by a person reading it again. Vantage automates the part of that loop a machine can do reliably and reserves a human for the part it cannot, which is judgment.

01
Deterministic Builds, Auditable to the Cell

Every figure is produced by code that returns the same answer on the same inputs. Run the same package twice and the numbers are identical, because nothing in the path depends on sampling, on temperature, or on a model's judgment. Open any workbook and trace a number back to its source. Nothing is a black box, and nothing depends on a model's judgment to be right.

02
A Deterministic Gate First, an Adversary Second

The deterministic checks run automatically on every build, and a build does not leave the shop until it clears them. They test structure rather than opinion, things like whether every tab that should exist exists, whether any cell carries an error value, whether the levers are wired to the schedules they claim to drive. On top of that, a separate review pass whose only job is to find what is wrong with the package runs before anything is released, handled by a reviewer that did not build it.

03
Your Numbers Never Run Through an AI Model

AI built the system. The figures and the commentary in your deliverables are both produced by deterministic code, and no model computes, alters, or writes a number in what you receive. The narrative in a board pack is templated off the same pipeline figures the tabs are built from, so the words and the numbers cannot drift apart.

Check It Yourself

What to do with the file when it lands

Every deliverable arrives as a formula-driven, self-contained workbook. That is a deliberate design choice, because a package you cannot check is a package you have to believe. Three things to try in the first five minutes.

Step One
Click Any Number

Pick the figure you trust least and click it. The formula bar shows the equation that produced it and the cells it points at. There is no hardcoded value sitting where a calculation should be, and the QC gate fails a build that ships one.

Step Two
Follow It Down to Source

Keep going. The schedule points at the register, the register row points at the account it came from, and the account points at the trial balance line exactly as you exported it. The trace ends where your own export begins, and nothing sits underneath it.

Step Three
Change an Input

Reject an add-back, move an assumption, and watch the schedules move with it. The input cells are wired to the outputs, which the gate verifies on every build, so the package behaves like a model you own rather than a picture of one.

The Limits

What the engine refuses to do

A verification story is only credible if it includes the part where the machine stops. These two behaviors are designed in, and they are the reason the rest of the package can be taken at face value.

01
It will not size what it cannot prove

Adjustments that need a market input a machine has no business inventing are identified, evidenced, and handed over unsized, with the basis written in words. Owner compensation is the standing example. The machine finds it and refuses to price it, because pricing it is a judgment call and judgment calls belong to a person who will answer for them.

02
It will not approximate around missing data

When a check depends on data that did not arrive, the workbook renders that check as pending and says which input is missing. Nothing is silently skipped and nothing is filled with an estimate that later reads as a fact. A month that cannot be reconciled shows up as a month that cannot be reconciled.
